# Project Architecture

Quick mental model: **`app/` = thin routes, `features/<x>/` = domain, `lib/` = generic infra, `db/` = persistence.**

## Decision tree — where does this file go?

```
Is it a route (page, layout, route handler, error.tsx, loading.tsx)?
  → app/(<group>)/<route>/<file>.tsx|ts
Is it domain logic tied to one product capability (auth, users, coffee, billing)?
  → features/<capability>/<slice>/<name>.<role>.ts
Is it generic, framework-agnostic infra (db client, http, auth provider, cn)?
  → lib/<area>/<file>.ts
Is it Drizzle schema?
  → db/schema/<feature>.ts (one file per owning feature) → re-exported via db/schema/index.ts
Is it Drizzle queries / data fetching?
  → features/<capability>/_queries/<name>.query.ts   (db/ never holds queries)
Is it a migration file?
  → db/migrations/<file>.sql  (generated by drizzle-kit, never hand-edited)
Is it a UI primitive (shadcn base-ui component)?
  → components/ui/<kebab>.tsx   (shadcn CLI manages this)
Is it env var validation / process.env access?
  → lib/env.ts (server-only, zod, exports `env` + `publicEnv`)
```

## Slice folders inside a feature

| Folder | File pattern | Role |
| --- | --- | --- |
| `_actions/` | `<name>.action.ts` | Server Actions |
| `_schemas/` | `<name>.schema.ts` | Zod schemas + inferred types |
| `_queries/` | `<name>.query.ts` | Data fetching (server-only) |
| `_hooks/` | `<name>.hook.ts` | React hooks (client) |
| `_types/` | `<name>.type.ts` | Domain types |
| `_utils/` | `<name>.util.ts` | Feature-local helpers |
| `_config/` | `<name>.config.ts` | Static config (routes, enums) |
| `_components/` | `<name>.tsx` | UI (kebab file, **no** role suffix) |

`_` prefix = private to the feature. Never deep-import across features.

## Component export style

- **Arrow functions only**: `export const Foo = (...) => { ... }`. Never `export function Foo() {}`.
- **Named exports only**: no `export default`. The barrel `index.ts` re-exports explicitly.
- **Export name**: PascalCase. File name kebab-case. `login-form.tsx` exports `LoginForm`.
- **kebab-case everywhere** (files, folders): Linux is case-sensitive; kebab avoids `Foo.tsx` vs `foo.tsx` collisions.
- **Subfolders in `_components/`**: flat by default. Promote to subfolders **only** when a component group grows genuinely complex (e.g. a reactive multi-section form: `components/form/sections/inputs/...`). No pre-emptive grouping. Barrel the subfolder if it has 3+ siblings.

## shadcn priority rule

**Always check the shadcn registry first** before writing a UI component. Boilerplate ships shadcn (`components.json` → `base-maia` style on `@base-ui/react`).

- Install with: `bunx --bun shadcn@latest add <component>` → lands in `components/ui/<kebab>.tsx`.
- `components/ui/` is shadcn-owned; don't hand-edit. To customize, copy into `features/<x>/_components/` and edit the copy.
- If shadcn covers the need, use it. Build a custom component only when the primitive falls short.
- Common primitives: `button`, `input`, `field`, `form`, `select`, `dialog`, `dropdown-menu`, `sonner`, `card`, `table`, `tabs`, `sheet`, `popover`, `tooltip`, `avatar`, `badge`, `checkbox`, `radio-group`, `switch`, `textarea`, `label`, `separator`, `skeleton`, `spinner`, `command`, `combobox`, `alert`, `alert-dialog`, `accordion`, `collapsible`, `breadcrumb`, `pagination`, `navigation-menu`, `sidebar`, `chart`.

## Barrels — split by boundary

Each feature exposes:

- `features/<x>/index.ts` — **client-safe**: types, schemas, components, hooks, utils
- `features/<x>/server.ts` — **server-only**: actions, queries. First line: `import "server-only"`

Import rules:
- Other features / routes: `import { Foo } from "@/features/<x>"` or `"@/features/<x>/server"`
- Deep imports into `_<slice>/` are a smell — surface them via barrel.

## app/ is a routing surface only

- Top-level route groups split by **auth gate**, not by feature:
  - `app/(public)/` — no session required (marketing, pricing, auth flow).
  - `app/(private)/` — session required (dashboard, settings, app surfaces).
  - `app/api/` — route handlers, no group. `api/auth/[...all]/route.ts` stays public.
- `app/(<group>)/<route>/page.tsx` is a shell: 5–20 lines, calls the feature, renders.
- `app/(<group>)/<route>/_components/` is **route-private UI only** (breadcrumbs, page-specific chrome). If reused → move to `features/<x>/_components/`.
- No business logic in `page.tsx`: no `await db.insert(...)`, no zod parsing, no fetch. All of that lives in the feature.

## Auth boundary — `proxy.ts`

- Single source of truth for public vs private path decisions (Next 16 middleware, Edge runtime).
- **Whitelist** of public prefixes: `/`, `/pricing`, `/login`, `/register`, `/forgot-password`, plus `api/auth/*`, `_next/*`, static assets. Anything not listed = private.
- Redirects: no session + private → `/login?next=<path>`. Session + `/login` or `/register` → `/dashboard`. Honor `next=` post-login.
- Read session via `auth.api.getSessionCookie()` (Edge-safe). **Never** import `lib/db` from proxy — Drizzle/libsql breaks on Edge.
- Matcher: exclude `_next/static`, `_next/image`, `favicon.ico`, `api/auth/*`.
- `app/(private)/layout.tsx` re-checks the session server-side as defense in depth.

## Boundaries — error / loading / not-found

- **Granularity**: global + per-group fallback. Add per-route only when a route truly needs distinct UX.
  - `app/error.tsx` — global error boundary (client, uses `reset`).
  - `app/global-error.tsx` — catches root-layout errors, must define its own `<html><body>`.
  - `app/not-found.tsx` — global 404.
  - `app/(public)/error.tsx`, `app/(public)/not-found.tsx` — public-context UX.
  - `app/(private)/error.tsx`, `app/(private)/not-found.tsx` — dashboard-context UX.
- **Loading**: hybrid. `loading.tsx` for route transition; `<Suspense>` inside the feature for sub-pieces that fetch in parallel.
- **Server Action errors**: discriminated union, never thrown across the boundary.
  - `Promise<{ ok: true; data: T } | { ok: false; error: string; fieldErrors?: Record<string, string[]> }>`
  - Client consumes with `useActionState` (React 19) for form/field errors. Unexpected errors → global toast.

## Form handling + validation flow

- **Library**: `useActionState` (React 19) nativo. No `react-hook-form` / `conform-to-zod` por default. Upgrade solo si un form lo demanda.
- **Pipeline**:
  1. Zod schema en `features/<x>/_schemas/<name>.schema.ts`. Client-safe (puede vivir en el barrel `index.ts`).
  2. Server action en `_actions/<name>.action.ts` (marcado `import "server-only"`): `schema.safeParse(input)` → si falla, return `{ ok: false, error, fieldErrors }`. Si pasa, ejecuta y return `{ ok: true, data }`.
  3. Client: `useActionState(action, initialState)` + render de `fieldErrors[name]` por field.

## lib/ vs features/

- `lib/` = no domain knowledge (auth config, http client, `cn`, env validation).
- `features/<x>/` = knows about a product capability.
- **No `features/_shared/`** — that's a junk drawer. If two features need the same thing: it belongs in `lib/` (if generic) or in a new feature owned by the domain.

## db/ (persistence)

- `db/` is the **only** place that knows about Drizzle/libsql. Exports: schema, drizzle client, seed scripts.
- **Schema by feature**: `db/schema/auth.ts`, `db/schema/users.ts`, `db/schema/billing.ts`. Barrel `db/schema/index.ts` re-exports every table. `drizzle.config.ts` reads from the barrel.
- better-auth-managed tables (`user`, `session`, `account`, `verification`) live in `db/schema/auth.ts`.
- **Queries belong to features**, not to `db/`. Use `features/<x>/_queries/<name>.query.ts`.
- **Edge runtime**: `db/index.ts` uses Node APIs (libsql). Never import `db/` from `proxy.ts` or any Edge file. Use `auth.api.getSessionCookie()` for session checks on the edge.

## Tests

- **Location**: colocated for unit/integration; `e2e/` at the repo root for end-to-end.
  - `features/<x>/_actions/login.action.test.ts` sits next to `login.action.ts`.
  - `e2e/<flow>.e2e.test.ts` for Playwright flows.
- **Naming**: double suffix — `login.action.test.ts`, `user.schema.test.ts`. Greppable by `rg "\.action\.test\.ts"`.
- **Runner**: Vitest (`vitest.config.ts`, `vitest.setup.ts`). E2E: Playwright (`playwright.config.ts`).
- **Component testing**: skip by default. Cover component behavior with E2E, not RTL.

## Anti-patterns

- `features/_shared/` — forbidden.
- `import { login } from "@/features/auth/_actions/login.action"` — bypasses the barrel; surface it in `index.ts` or `server.ts`.
- `app/(dashboard)/users/_actions/create-user.action.ts` — domain logic in the routing tree. Move to `features/users/_actions/`.
- `page.tsx` > 50 lines, contains DB calls or zod parsing — feature boundary violated.
- `features/auth/components/` (no underscore) — breaks the private-folder convention. Always `_components/`.
